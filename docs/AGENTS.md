# `docs/` 代理指引

## 目的

`docs/` 是公開旅遊指南的內容來源。頁面以台灣繁體中文讀者為對象，並依地點與內容類型組織。

## 目錄與檔名

- 使用現有層級：`國家/地區/資料類型/頁面.md`，例如 `日本/東京/景點/東京晴空塔.md`。
- 有些地區會使用更深的層級，例如 `國家/縣市/區域/景點/景點名稱.md`。編輯目的地時，請遵循鄰近既有結構。
- 國家、地區或分類總覽頁使用 `index.md`。
- 城市或地區總覽頁通常應同時有 `index.md` 與 `index.schema.json`，尤其是鄰近地區已採用此模式時。
- 常見內容類型資料夾包含 `景點`、`活動`、`_交通`、`_注意事項`、`_其他` 與 `_旅遊須知`。
- 開頭底線用於將通用或跨分類主題排序到導覽前方。除非主題刻意不是一般旅遊分類，否則不要新增底線資料夾。
- 保留現有中文路徑名稱。不要只為了統一措辭而重新命名檔案。

## 寫作風格

- 使用繁體中文，語氣應務實、像旅遊指南。
- 內容主張應具體且有用：位置、交通、亮點、旅人提示、注意事項與官方連結。
- 避免無來源或過度宣傳的文字。
- 優先提供旅人需要的事實：交通方式、票券或預約規則、營業時間、官方公告與官方連結。主觀旅遊心得應放在次要位置。
- 對於易變動資訊，例如營業時間、票價、預約規則、休館、交通路線、簽證規定與活動日期，修改前必須確認最新官方來源。
- 研究台灣以外目的地時，可行的話先用當地官方語言搜尋，再選擇正確的官方繁體中文頁面；若沒有繁中頁，使用最具權威性的官方入口。
- 不要從來源複製長段文字。請用原創文字摘要，並以連結引用來源。

## 頁面結構

- 頁面開頭使用單一 `#` 標題，並與頁面主題一致。
- 主要主題使用 `##` 區塊。既有頁面常見區塊包含 `地圖`、`交通指南`、`必訪景點`、`鄰近景點推薦`、`溫馨提示` 與 `相關連結`。
- 景點頁面應在適用時涵蓋簡介、景點特色、門票或預約資訊、交通方式、地圖與官方相關連結。
- 外部參考資料放在 `## 相關連結`，並使用 Markdown 連結。
- 相關內部連結保持相對路徑且省略副檔名，例如 `../_交通/index` 或 `./淺草寺`。
- Markdown 連結優先使用可讀的非編碼路徑，例如 `[淺草寺](./淺草寺)`，不要把中文或日文字元轉成百分比編碼。
- 鄰近景點推薦應優先連到既有頁面，並確認目標路徑存在。
- 主動建立相關頁面之間的連結，避免資訊孤島：景點連到交通指南、地區連到鄰近地區、活動頁連到相關目的地頁。
- 保持版面整潔。避免重複分隔線與過多連續空白行。

## 圖片與嵌入

- 不要在 `docs/` 下新增本地圖片檔。
- 只有在授權與穩定性可接受時，才使用外部託管圖片網址。頁面需要視覺內容時，優先直接嵌入圖片，而不是連到圖片庫頁面。
- 可使用 AI 生成示意圖，但必須確認授權安全且適合頁面；不要將其呈現為紀實照片。
- Wikimedia Commons 直接圖片網址可用 `npx tsx scripts/get-wikimedia-image.ts "File:Example.jpg"` 取得；若環境有 Bun，也可使用 `bun scripts/get-wikimedia-image.ts "File:Example.jpg"`。
- Google Maps iframe 應放在 `## 地圖` 下，包含 `src`、`loading`、`allowfullscreen`、`referrerpolicy` 等功能性屬性，並省略固定尺寸與 inline style。

## Schema 檔

- 若 Markdown 頁面有對應的 `*.schema.json`，請讓 schema 與可見頁面內容保持同步。
- 地點頁適合時使用 Schema.org `Place`，並包含準確的 `name`、`description`、`address` 與 `url`。
- 城市或地區總覽頁建立 `index.schema.json` 時，使用適合的 Schema.org 類型，例如 `City` 或 `AdministrativeArea`。
- Schema URL 應對應 VitePress clean URL，且不包含 `.md`。
- Schema URL 中的中文與日文字元保持未跳脫；只有符號必要時才編碼。
- 移動或重新命名頁面後，不要留下過期 schema 檔。

## 驗證

- 大量修改 prose 或 Markdown 後，可行的話執行 `npm run correct` 與 `npm run format`。
- 內容修改後執行 `npm run check`。
- 新增頁面、修改會影響標題的 heading、編輯 schema JSON 或修改內部連結時，執行 `npm run build`。
- 不要在 VitePress config 啟用 `ignoreDeadLinks: true` 來繞過壞連結。請修正路徑，或新增預期的佔位頁。
- 新增或修改外部連結時，使用 `npx tsx scripts/verify-links.ts path/to/file.md` 針對 `相關連結` 做聚焦檢查。
- 新增或修改外部連結時，可行的話手動確認官方 URL 與最終重新導向目標，例如 `curl -I -L -A "Mozilla/5.0" URL`；自動檢查可能漏掉重新導向或錯誤入口頁。
